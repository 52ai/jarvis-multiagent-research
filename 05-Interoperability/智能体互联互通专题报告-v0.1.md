## 📋 摘要

当 AI 智能体从"封闭世界"（单一平台内部协作）走向"开放世界"（跨平台、跨组织协作），互联互通成为决定其能否规模化的**最关键基础设施**。本报告从三个层次——**业务层、协议层、网络层**——系统梳理 2025-2026 年间涌现的智能体互联互通方案，并从**发展与监管**双重视角，给出产业、政策、技术的三维推进建议。

**核心论点**：
1. 智能体互联互通不是单一协议问题，而是**三层架构问题**（业务/协议/网络）
2. **互联网基础设施的演化模式正在被精准复制到智能体层**（DNS → ANS；BGP → AIN；HTTP → MCP/A2A）
3. **2026 年是"智能体互联网"基础设施窗口期**——错过了将形成 5-10 年的标准化锁定

## 一、问题陈述：为什么"互联互通"是智能体规模化的瓶颈

### 1.1 现状：智能体"封闭世界"问题

2025-2026 年间，几乎所有主流 AI 平台都发布了智能体框架：
- **Anthropic** — Claude with Tools
- **OpenAI** — GPTs, Operator
- **Google** — Gemini Agents, A2A
- **Microsoft** — Copilot Agents
- **Meta** — Llama Agents

但这些智能体**默认只能在自家平台内协作**。一个 Claude 智能体无法直接发现并调用一个 Gemini 智能体——除非事先通过自定义 API 网关做"硬编码对接"。

**N² 双边集成问题**：
- 5 个智能体平台 → 25 条独立对接
- 10 个智能体平台 → 100 条对接
- 100 个平台 → 10,000 条对接
- 这正是 1983 年互联网"hosts 文件时代"的复刻

### 1.2 互联互通缺位的真实代价

| 代价维度 | 现状 | 互联互通后的预期 |
|---------|------|------------------|
| **开发成本** | 每对接一个新平台 = 4-6 周工程 | 标准协议下 1-2 天 |
| **创新速度** | 闭源生态，新进入者壁垒高 | 开源协议，无门槛接入 |
| **跨组织协作** | 需要中心化平台撮合 | 任意智能体间直接 P2P |
| **监管可见性** | 各平台黑盒，无统一审计 | 协议层天然支持可观察性 |

### 1.3 历史镜鉴

| 互联网时代 | 关键基础设施 | 解决的核心问题 | 标准化窗口 |
|----------|------------|--------------|-----------|
| 1983 | DNS | 跨网络主机命名 | 1983-1987 |
| 1989 | BGP | 跨 AS 路由 | 1989-1994 |
| 1995 | SSL/TLS | 跨网络加密通信 | 1995-1999 |
| 1996 | XML/SOAP | 跨系统数据交换 | 1996-2003（被 JSON/REST 取代）|
| **2026** | **智能体协议簇** | **跨平台智能体协作** | **2025-2028（窗口期）** |

## 二、三层架构全景

智能体互联互通可以分解为三个独立但相互依赖的层次：

```
┌────────────────────────────────────────────────────────┐
│  业务层 (Business Layer)                                │
│  ─────────────────────────────────────                  │
│  关注：智能体间的商业关系、交易、契约、合规                │
│  协议：ACP (IBM) · UCP (Google) · NANDA Index          │
├────────────────────────────────────────────────────────┤
│  协议层 (Protocol Layer)                                │
│  ─────────────────────────────────────                  │
│  关注：智能体如何"说同一种语言"、交换消息                 │
│  协议：MCP (Anthropic) · A2A (Google) · AIN (IETF)     │
├────────────────────────────────────────────────────────┤
│  网络层 (Network Layer)                                │
│  ─────────────────────────────────────                  │
│  关注：智能体如何被找到、被信任、被路由                    │
│  基础设施：ANS · DNS-AID · .Agent · AgentDNS · AGNTCY  │
└────────────────────────────────────────────────────────┘
   ↓ 都建立在现有互联网基础设施之上（TCP/IP, DNS, BGP, TLS）
```

### 2.1 业务层（Business Layer）

**定义**：处理智能体间**商业关系**的协议层——交易、定价、支付、合同、争议解决。

| 协议 | 主体 | 关键特性 | 2026 状态 |
|------|------|---------|----------|
| **ACP** (Agent Commerce Protocol) | IBM Research → Linux Foundation | 开放、厂商中立；词汇含定价、报价、支付确认、协商 | 1.0 规范发布 |
| **UCP** (Universal Commerce Protocol) | Google | 嵌入 Google Shopping 图谱的智能体商务 | 商用 |
| **NANDA Index** (MIT Raskar) | MIT | 跨平台智能体发现 + 已验证 AgentFacts | 研究中（arXiv:2507.14263）|

**业务层的 4 个核心问题**：
1. **定价** — 智能体服务的"价格"如何表达？
2. **支付** — 智能体之间能直接结算吗（机器对机器支付）？
3. **责任** — 智能体 A 调用智能体 B 出错，谁负责？
4. **合规** — 跨境智能体交易如何满足 GDPR / AI Act？

### 2.2 协议层（Protocol Layer）

**定义**：智能体**交换消息**的协议层——和 HTTP 在 Web 中的地位类似。

| 协议 | 主体 | 关键特性 | 2026 状态 |
|------|------|---------|----------|
| **MCP** (Model Context Protocol) | Anthropic（开源）| 智能体↔工具/数据；JSON-RPC；schema 发现 | 9700 万次下载；几乎全行业支持 |
| **A2A** (Agent2Agent) | Google → Linux Foundation | 智能体↔智能体；Agent Card；任务委派 | 150+ 组织；落地 3 大云 |
| **AIN** (Agentic Intent Network) | IETF NMRG | 智能体间路由；类 BGP/IP | 草案阶段（2026-04）|

**协议层的关键设计选择**：
- **A2A vs MCP 边界**：MCP 解决"智能体如何使用工具"，A2A 解决"智能体如何与其他智能体协作"
- **AIN 定位**：是"智能体层的 IP"——智能体如何被路由到，而不是用哪个 API 调用

### 2.3 网络层（Network Layer）

**定义**：智能体的**身份、命名、发现、验证**层。

| 方案 | 主体 | 命名机制 | 身份验证 | 发现机制 |
|------|------|---------|---------|---------|
| **ANS v2** | OWASP GenAI → IETF | DNS 域名 | DANE + TLSA | DNS + ANS registry |
| **DNS-AID** | IETF (Mozley et al.) | DNS 域名 | DNSSEC + SVCB + DANE | DNS SVCB 查询 |
| **.Agent** | Headless Domains | 顶级域 `.agent` | DNSSEC | DNS |
| **AgentDNS** | arXiv:2505.22368 | DNS 根域 | DNSSEC | DNS 根区 |
| **AGNTCY** | Cisco + LF | 中心化 ID | 证书链 | 中心化目录 |
| **DNSid** | IETF (Ihsanullah) | DNS | DNSSEC | DNS |

**网络层的 4 个核心问题**：
1. **命名** — 智能体需要一个全球唯一的名字（类似域名/IP）
2. **身份** — 这个智能体是"谁"？它声称的能力是真的吗？
3. **发现** — 给定一个智能体名，怎么找到它的接入点？
4. **信任** — 它提供的服务能信吗？历史表现如何？

### 2.4 三层关系与依赖链

```
应用层
   │
   ▼ 调用
协议层（A2A / MCP / AIN）
   │  寻址
   ▼ 查询
网络层（ANS / DNS-AID / AGNTCY）
   │  解析
   ▼ 映射
物理层（IP / DNS / BGP / TLS）
```

**类比**：
- 业务层 = 电子商务（Amazon、Taobao）
- 协议层 = HTTP（Web 的统一协议）
- 网络层 = DNS + IP（互联网的"地址簿 + 路由"）

## 三、典型方案深度对比

### 3.1 协议层：MCP vs A2A vs AIN

| 维度 | MCP | A2A | AIN |
|------|-----|-----|-----|
| **目标** | 智能体↔工具 | 智能体↔智能体 | 智能体↔智能体（路由视角）|
| **抽象** | 工具调用 | 任务委派 | 能力路由 |
| **状态** | 有状态（连接会话）| 短状态（任务级）| 无状态（数据报）|
| **拓扑** | 客户端-服务器 | 多对多 | 多对多（路由发现）|
| **发现机制** | schema 自描述 | Agent Card | CAP（Capability Advertisement）|
| **可类比** | ODBC / JDBC | SMTP | BGP / IP |

**关键洞察**：
- **MCP 是"垂直整合"**——一个智能体如何使用外部资源
- **A2A 是"水平协作"**——智能体如何分工
- **AIN 是"基础设施层"**——智能体如何被寻址

**未来格局预判**（基于 v1.0 规范成熟度）：
- **MCP** 大概率成为**工具层事实标准**（已 9700 万下载）
- **A2A** 大概率成为**智能体间协作默认协议**（Linux Foundation 治理）
- **AIN** 仍处草案，3-5 年后可能成为"智能体网络的 IP"

### 3.2 网络层：ANS vs DNS-AID vs AGNTCY

| 维度 | ANS v2 | DNS-AID | AGNTCY |
|------|--------|---------|--------|
| **架构** | 联邦（DNS + 外部 registry）| 纯 DNS | 中心化目录 |
| **Lookup 次数** | 多次（DNS + DANE + registry）| 1 SVCB + 1 DANE | 1 HTTPS |
| **延迟** | 100ms-1s（多 RTT）| 1-10ms（单 UDP）| 50-200ms（HTTPS）|
| **信任模型** | DANE + ANS 治理 | DNSSEC + DANE | PKI + 平台信用 |
| **去中心化程度** | 中 | 高 | 低 |
| **可扩展性** | 高（继承 DNS）| 极高 | 中（中心化瓶颈）|

**Verisign 2026-06 实证结论**：
- 智能体 P90 元数据 = **940 字节**（DNSSEC + DANE + 能力描述）
- 单次 DNS UDP 响应上限 = 1232 字节
- **结论：完全可塞进单条 DNS 消息**，无碎片化延迟

**未来格局预判**：
- **DNS-AID / ANS** 路线（基于 DNS）将占主导——**继承 40 年互联网基础设施**
- **AGNTCY** 中心化模式适合"封闭花园"场景（如企业内部）
- **混合模式**最可能：DNS-AID 做发现，AGNTCY 做推荐

### 3.3 业务层：ACP vs UCP vs NANDA Index

| 维度 | ACP | UCP | NANDA Index |
|------|-----|-----|-------------|
| **生态** | 开放（Linux Foundation）| Google 闭环 | MIT 研究 |
| **交易对象** | 任意 B2B | Google Shopping | 不限（带声誉评分）|
| **支付** | 任意支付方式 | Google Pay | 任意 |
| **声誉系统** | 暂无 | Google 信用 | AgentFacts（已验证）|

**业务层的"无人区"**：
- 智能体间**责任归属**几乎无标准
- 跨境智能体交易**税务**问题无先例
- 智能体**保险**市场尚未启动

## 四、Verisign 实证：智能体元数据天然适合 DNS

> **来源**：Seethiraju, Thakar, Shyamsunder, Osterweil (Verisign) · arXiv:2606.02314v1 · 2026-06-01

### 4.1 三维评估框架

1. **Navigational Completeness** — 元数据是否包含可定位性、能力、协议、真实性
2. **Lookup Complexity** — 解析需要的查询次数
3. **Transaction Performance** — UDP ms 级 vs TCP 多 RTT

### 4.2 119,757 端点测量结果

| 指标 | 中位数 | P90 |
|------|--------|-----|
| 端点 URL 长度 | 70 B | 108 B |
| MCP 工具数/server | 3 | 19 |
| 工具名长度 | 16 字符 | 26 字符 |
| 能力命名空间 footprint | 49 字符 | 323 字符 |
| **完整 P90 元数据** | — | **609 B** |
| + DNSSEC (RSA-2048) | — | +296 B = 905 B |
| + DANE SHA-256 | — | +35 B = **940 B** |

### 4.3 三个对 033 课题的启示

1. **智能体互联互通的"物理层"已经被互联网解决**——DNS + DNSSEC + DANE 全部可用
2. **元数据规模天然适配 DNS UDP 消息**——无需发明新传输协议
3. **"智能体互联网"不应从零建基础设施**——而应**复用** 40 年互联网基础设施

## 五、互联网基础设施的"四十年复制"

下表展示了**互联网基础设施演化**与**智能体基础设施演化**的精确对应关系：

| 互联网（人类）| 智能体（AI）| 解决的核心问题 | 标准化主体 |
|------------|------------|--------------|----------|
| DNS（1983-）| ANS / DNS-AID / .Agent（2025-）| 命名 + 寻址 | IETF / ICANN |
| BGP（1989-）| AIN / AgentDNS（2026-）| 跨域路由 | IETF |
| TLS（1995-）| DANE / 智能体身份验证（2026-）| 加密 + 认证 | IETF |
| HTTP（1996-）| MCP / A2A（2024-）| 应用协议 | IETF / Linux Foundation |
| SMTP（1982-）| 智能体间消息 / 通知协议（2026-?）| 异步消息 | 待定 |
| SOAP（1996-）→ REST（2000-）| MCP（2024-）| 工具调用 | Anthropic → 全行业 |
| 信用卡支付（1950-）| ACP / UCP（2026-）| 商业交易 | IBM / Google |
| DNS WHOIS（1982-）| NANDA AgentFacts（2025-）| 智能体事实查询 | MIT |
| 域名注册商（1993-）| ANS Registry（2026-）| 命名注册 | OWASP / IETF |

**核心洞察**：
> **智能体互联网不是要"发明新互联网"，而是要"在 40 年互联网基础设施之上插入智能体层"**——和 Web（HTTP/HTML）在 TCP/IP 上面的关系一样。

**这一洞察的政策含义**：
- 监管者应**优先支持基于现有互联网标准的方案**（如 ANS/DNS-AID）
- 反对**封闭花园式协议**（如某些厂商私有协议）
- 推动**关键命名空间（如 `.agent`）进入 ICANN 议程**

## 六、产业/政策/技术推进建议

### 6.1 产业建议（面向企业、平台、开发者）

#### 短期（6-12 个月）
1. **采用 MCP 作为工具层标准**——已 9700 万下载，是事实标准
2. **企业内 A2A 试点**——跨部门智能体协作（如销售智能体调用财务智能体）
3. **关注 Linux Foundation 治理动态**——A2A 治理权已移交 LF

#### 中期（1-2 年）
4. **企业内部部署 ANS / DNS-AID**——给企业智能体发"内部域名"
5. **建立智能体能力市场**——类似苹果 App Store 模式
6. **智能体责任险**——为智能体调用提供保险产品

#### 长期（3-5 年）
7. **行业垂直智能体协议**——金融、医疗、法律等领域的特殊需求
8. **跨链 / 跨云智能体迁移**——避免厂商锁定
9. **智能体间经济模型成熟**——机器对机器支付 / 微支付成主流

### 6.2 政策建议（面向监管者、政府、国际组织）

#### 1. 推动 ICANN 议程纳入智能体命名（紧迫）
- 申请 `.agent` 顶级域进入根区
- 参考 2014 年 `.bank` 金融专用域的模式
- 由 ITU / ICANN / W3C 联合治理

#### 2. 制定智能体互联互通强制标准（参考 EU AI Act）
- 2026-08-02 EU AI Act 全面实施，可**扩展到智能体间通信**
- 强制要求：智能体能力描述可机器读取、智能体间通信可审计
- 中国可参考：制定《智能体互联互通管理办法》

#### 3. 建立智能体身份认证体系
- 数字证书 + DNSSEC 链验证
- 跨境互认：参考 eIDAS 电子身份框架
- 实名 / 匿名分层：公共智能体强制实名，私有智能体可选

#### 4. 智能体责任归属的法律框架
- **核心原则**：智能体的法律行为归属其**运营者**，技术故障归属**开发者**
- 跨境智能体交易的司法管辖
- 智能体保险与赔偿基金

#### 5. 反垄断与互操作性
- 禁止平台强制封闭（如要求只能用 A 平台智能体）
- 参考欧盟《数字市场法》(DMA) 强制互通条款
- 防止"智能体操作系统"垄断

#### 6. 安全与隐私监管
- 智能体间通信加密强制（类 GDPR）
- 智能体日志保留（用于审计）
- 智能体跨境数据传输（数据出境安全评估）

### 6.3 技术建议（面向研究机构、标准组织、开源社区）

#### 1. 加速关键标准成熟
- **IETF**：推动 AIN 草案 → RFC（目标 2027）
- **IETF**：合并 DNS-AID / ANS / DNSid 为统一标准
- **Linux Foundation**：A2A 1.0 长期支持（LTS）规范

#### 2. 重点研究课题
- **智能体元数据的语义互操作**——能力描述统一本体论
- **跨协议网关**——MCP↔A2A、A2A↔AIN 转换层
- **智能体声誉系统**——去中心化信誉评分（DID + 区块链）
- **智能体安全验证**——形式化验证智能体协议
- **智能体间经济机制**——博弈论 + 机制设计

#### 3. 开源生态扶持
- 资助参考实现（如 ANS、AIN、AGNTCY）
- 建设公共智能体测试床（testbed）
- 互操作性认证计划（类似 Wi-Fi 联盟）

#### 4. 性能与可扩展性
- 智能体发现的 QPS 优化
- 智能体路由的延迟优化
- 大规模智能体网络的稳定性测试

#### 5. 学术研究开放问题
| 问题 | 当前研究空白 | 潜在价值 |
|------|------------|---------|
| 智能体语义互操作 | 各协议能力描述不统一 | 跨平台智能体迁移 |
| 智能体信任传播 | 缺乏去中心化信任机制 | 抗攻击的智能体网络 |

## 七、未解问题与未来研究方向

### 7.1 短中期关键问题（1-2 年内）
- **Q1：A2A 与 MCP 的"边界争议"如何解决？**——谁处理"智能体调用智能体但需要工具"场景？
- **Q2：智能体间身份是否会引发"实名 vs 匿名"分裂？**——类 Tor 网络 vs ICANN 实名
- **Q3：智能体交易如何征税？**——美国 / EU / 中国可能采用不同模式
- **Q4：智能体责任归属是否承认"智能体法人"？**——日本 / 韩国可能率先试点

### 7.2 长期开放问题（3-5 年）
- **Q5：智能体互联网是否会形成"国家网"分裂？**——类早期互联网分裂
- **Q6：智能体间是否能形成真正的"通用语"？**——LLM 是否会演化出标准通信语
- **Q7：智能体集群是否会产生"涌现智能"？**——比单智能体强多少
- **Q8：智能体互联互通是否会带来新的"系统性风险"？**——如智能体级联故障
